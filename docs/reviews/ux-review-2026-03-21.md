# UX Designer Review — 2026-03-21

**Reviewer perspective:** UX designer evaluating user flows, cognitive load, interaction patterns, and overall usability.

---

## User Flow Analysis

### Primary Journey: "I want to see what's happening with my stocks"

```
Open app → Feed (My Watchlist) → Scan cards → Click event → Read detail → Back to feed
```

**Friction points:**
1. **3-5 second skeleton loading** before feed renders. For a "real-time" tool, this is an eternity. Users see gray bars and don't know if it's loading or empty.
2. **Clicking an event navigates to a full-page detail** on mobile. The back button works, but you lose your scroll position in the feed. On desktop, the detail panel appears on the right (good), but it still triggers a URL change to `/event/:id`.
3. **No way to quickly scan multiple events.** You have to click into each one to see the AI analysis. Consider showing the BEARISH/BULLISH verdict and a one-line thesis directly on the card (you do show the badge, but not the "why").

### Secondary Journey: "Set up my alerts"

```
Onboarding → Pick tickers → Skip notifications → Feed
(Later) Settings → Enable push → Permission denied → ???
```

**Friction points:**
1. If the user denied notifications during onboarding ("Maybe later") and later wants to enable them, the Settings page shows "PERMISSION: DENIED" with no guidance on how to fix browser-level permission denial. The user is stuck.
2. The 3-step push setup guide (Tap Enable → Allow in prompt → Return to watchlist) is clear, but step 2 relies on the browser permission prompt which the headless test environment denied. There's no fallback: no "check your browser settings" link, no email alternative.

### Tertiary Journey: "Can I trust these signals?"

```
Feed → Click event → Scroll to Trust section → See "Pending" everywhere → Scorecard tab → See 24.4% hit rate → ???
```

**Friction points:**
1. The trust journey is actually well-intentioned. But the event detail page buries the trust info at the very bottom — users must scroll past 8+ sections to find it.
2. When trust data shows all "Pending" and "Insufficient Data," it doesn't build confidence. Consider hiding the trust section until data is available, or showing a message like "Tracking in progress — check back in 5 trading days."
3. The Scorecard tab on the event detail page is broken (duplicates the UI), so users literally cannot access per-event trust info through the tab interface.

## Cognitive Load — 6/10

### Event Detail Page: TOO MUCH

The event detail page has 11 distinct sections visible on scroll:

1. Header (severity, type, age)
2. Sidebar (Market Regime, Quick Actions)
3. Catalyst Summary (What Happened)
4. Market Context (Why It Matters Now)
5. Directional Thesis (Bull vs Bear)
6. Risk Factors
7. Source Details
8. Market Regime (duplicated from sidebar!)
9. Alert Provenance (Source Journey)
10. Trust/Scorecard Context (Verification)
11. Feedback + Disclaimer

This is information overload. A trader scanning breaking news doesn't need all 11 sections simultaneously. The tab structure (Summary/Evidence/Trust) was the right idea, but the tabs are broken — everything dumps onto the Summary tab.

**Recommendation:**
- Summary tab: Header + What Happened + Bull/Bear + Quick Actions. That's it.
- Evidence tab: Market Context + Source Details + Risk Factors
- Trust tab: Source Journey + Scorecard Verification + Market Regime

### Feed Page: JUST RIGHT

The feed has appropriate information density. Each card shows: severity, type, age, headline, directional badge, snippet, source, tickers. That's the right amount for scanning.

### Scorecard Page: TOO MUCH (but appropriately so)

The Scorecard is inherently data-heavy. The "How to read this" callout helps. But there are 15+ bucket tables on a single scroll. Consider collapsible sections (they seem to exist but are all expanded by default).

## Interaction Patterns — 7/10

**What works:**
- Feed filter chips (HIGH × , CRITICAL ×) with clear dismiss buttons
- Dropdown for sort order (Latest first / Highest severity)
- Star/bookmark on feed cards
- Ticker tag pills on cards linking to profiles
- Drag handles on watchlist for reordering
- Keyboard shortcuts (j/k navigation, / for search)

**What doesn't:**
- **The "Filters" button** opens an inline filter bar, not a dropdown or modal. The bar shows "Push alerts only" and "+ Add filter." But clicking "+ Add filter" — what options are there? The interaction model is unclear. What can I filter BY? Severity? Source? Ticker? Date range? The feed doesn't expose these options clearly.
- **The feed dropdown** (My Watchlist / All Events) is a non-standard control. Most apps use tabs or a segmented control for view switching, not a dropdown. Dropdowns imply many options; this has exactly two.
- **No pull-to-refresh on mobile.** For a real-time feed, users expect pull-to-refresh as a gesture.
- **No infinite scroll indicator.** The feed shows "Loading more..." at the bottom, which is fine, but there's no indication of how many total events exist or whether you've reached the end.

## Empty States — 5/10

**Tested scenarios:**

- **Feed detail (no event selected):** Shows a newspaper icon + "Select an event to view details / Click any alert card on the left." This is GOOD — clear instruction, appropriate illustration.
- **Feed (no events matching filters):** NOT TESTED — the current data always has events. But if a user's watchlist tickers have no events, what do they see? This needs a helpful empty state: "No events for your tickers yet. Events are checked every X minutes. In the meantime, explore trending tickers."
- **Bull Case empty:** Shows "No upside thesis identified." This is fine but cold. Consider: "Our analysis didn't find a bullish case for this event. That doesn't mean there isn't one — do your own research."

**Missing empty states:**
- What if the user removes all watchlist tickers? Does the feed still work?
- What happens at Scorecard with 0 verdicts?
- What does History look like with a date range that has no events?

## Error Handling UX — 4/10

**Push notification denial:** The Settings page shows "PERMISSION: DENIED" in a yellow/orange banner. But it doesn't explain:
- WHY it was denied (browser blocked it)
- HOW to fix it (browser settings → site permissions → notifications → allow)
- WHAT ALTERNATIVE exists (email, webhook, etc.)

**WebSocket failure:** Silent. The "Live" indicator shows a red dot suggesting the connection is active, but the WebSocket actually failed. There's no visual feedback that live updates are degraded.

**Network errors:** Not tested (all requests succeeded), but there's no visible error boundary or retry mechanism in the UI. If the API goes down, users likely see eternal skeleton loading.

---

## Progressive Disclosure — 6/10

**Good examples:**
- Onboarding gradually introduces concepts: tickers → notifications → scorecard explanation
- Feed cards show summary; click for full detail
- Settings sections are collapsible accordions
- Scorecard has a "How to read this" explainer

**Bad examples:**
- Event detail dumps all 11 sections at once instead of using tabs properly
- History page shows all filters immediately — date range, severity, source, ticker. For most users, the defaults are fine. Consider hiding the filter bar behind a "Filters" toggle (like the Feed does).
- Scorecard shows ALL bucket tables expanded. Signal buckets, confidence buckets, source buckets, event type buckets — that's 20+ data tables visible at once.

---

## Micro-interactions — 6/10

**Present:**
- Orange highlight on selected feed card (left border accent)
- Checkmark appears on selected ticker buttons in onboarding
- Hover state on nav items (orange underline)
- Skeleton loading animation (pulse effect)

**Missing:**
- No transition animation when opening event detail (it just appears)
- No toast/notification on successful actions (adding ticker to watchlist, submitting feedback)
- No loading state on "Useful/Not useful/Bad data" feedback buttons — user doesn't know if their feedback was received
- No hover tooltip on severity badges or confidence indicators
- No subtle animation on the "Live" indicator (a pulsing dot would reinforce liveness)

## Mobile Experience

### 375px (Mobile) — 7/10
- Feed cards stack well, readable text sizes
- Filter chips wrap to second line (appropriate)
- Bottom nav is clean, icons are recognizable
- Event detail is full-screen with back button — works
- **Issue:** Scorecard page never loads (skeleton only)
- **Issue:** Feed dropdown defaults to "All Events" instead of "My Watchlist"

### 768px (Tablet) — 8/10
- Single-column feed, no detail pane (appropriate for width)
- Cards have more breathing room
- All pages render correctly
- Clean, readable layout

### 1280px (Desktop) — 8/10
- Two-panel master-detail layout on Feed
- Scorecard renders fully with all bucket tables
- Watchlist uses full width effectively
- History filters sit naturally at top

## Top 5 UX Improvements (Ranked by User Impact)

### 1. Fix the event detail tabs and restructure content
The three tabs (Summary/Evidence/Trust) should each show DIFFERENT, focused content. Currently they're broken and everything appears on Summary. Proper tab separation would cut cognitive load by 60% per view.

### 2. Add a "What happened after this alert" follow-up mechanism
Right now, users see an alert and then... nothing. Close the loop. Show price movement post-alert. This is the single biggest gap in the user journey — you detect the event but never show the outcome.

### 3. Fix push notification UX with fallback channels
The notification flow is a dead end if the browser denies permission. Offer email digests, Discord webhooks, or Telegram as alternatives. A real-time alerting product without working notifications is an oxymoron.

### 4. Add pull-to-refresh and visible connection status
Mobile users need pull-to-refresh. All users need to know if the WebSocket is connected. Replace the silent "Live" dot with a connection indicator that changes state: "Live" (green) → "Reconnecting..." (yellow) → "Offline" (red).

### 5. Reduce feed loading time or add meaningful loading states
3-5 seconds of gray skeleton bars is too long. Either cache the last-seen feed for instant render, or replace skeleton with a purposeful loading state: "Checking 14 sources for your 10 watchlist tickers..."
