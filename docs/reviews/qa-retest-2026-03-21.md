## NEW Issues Found

### N1. "DUMMY" source still visible in Scorecard (Severity: Medium)
- **Page:** `/scorecard` → Source buckets section
- **What:** "DUMMY" appears as a source bucket with 2,185 alerts. While "dummy" was filtered from the Feed filter dropdown, the Scorecard backend API still returns dummy source data and the Scorecard page renders it.
- **Root cause:** The `/api/events/sources` endpoint still returns "dummy" in its list. Filtering was only applied on the frontend Feed filter component, not on the Scorecard or the API.
- **Impact:** Users see "DUMMY" as a source name in Scorecard, which looks like test/debug data leaked into production.

### N2. "?" Button navigates to Settings instead of showing shortcuts (Severity: Low)
- **Page:** All pages (top-right corner)
- **What:** The `?` icon button in the header is a link to `/settings`. Pressing the `?` keyboard key shows the keyboard shortcuts modal. This is a UX inconsistency — users clicking the visible `?` button expect the same behavior as pressing `?`.
- **Impact:** Minor UX confusion. The button and key do different things.

### N3. WebSocket reconnection loop (Severity: Low)
- **Page:** All pages
- **What:** `WebSocket connection to 'ws://localhost:5173/ws/events' failed: WebSocket is closed before the connection is established.` repeats every ~20 seconds. The client keeps retrying the WS connection.
- **Impact:** Console noise. No user-visible impact since the "Live" indicator shows and the feed loads via REST. In production with the WS server running, this would not be an issue. But the retry loop should have a backoff or max-retry limit.
